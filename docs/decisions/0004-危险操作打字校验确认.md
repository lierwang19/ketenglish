# 0004 — 危险操作用打字校验对话框替代 window.confirm

- **日期**：2026-04-26
- **状态**：✅ 生效中
- **影响范围**：vocab-review（listening-player 后续应同步）

## 背景

周次管理功能上线后，Codex 复核出 4 个 finding，其中 P1 是"清空整周 / 删除整周"用了 `window.confirm()`：

- 浏览器原生确认框样式难看，与 App 整体设计割裂
- "确认 / 取消"两个按钮太轻，家长可能误点确认就丢失整周数据
- 没有任何"二次校验"，连孩子都能轻易触发

被删除的可能是几十个单词 + 全部学习进度 + 全部复习日志，无法撤销。

## 选项

| 方案 | 优 | 劣 |
|---|---|---|
| A. 保持 window.confirm | 零开发 | UI 割裂、误操作风险高 |
| B. 自定义确认对话框（点击两个按钮） | UI 一致 | 仍然容易误点 |
| C. 自定义确认对话框 + **要求用户打字校验**（如输入"清空"两个字才能点确认） | 强制刻意行为，几乎不可能误操作 | 多一步操作 |

## 决定

选 **C：打字校验确认对话框**。

落地：

- `vocab-review/js/app.js` 新增 `confirmDanger({ title, message, requireText, confirmLabel })`
- 返回 `Promise<boolean>`：用户必须在输入框打出 `requireText`，确认按钮才解禁
- 支持 Esc 取消、Enter 提交、点击背景取消、自动 focus 输入框
- CSS 在 `vocab-review/css/main.css` 加 `.danger-modal-*` 系列样式
- 配套：异步按钮处理器用 `withBusy(btn, fn)` 包装，禁用按钮直到完成，防双击

使用规范（写入 [CONVENTIONS.md](../../CONVENTIONS.md)）：

> 任何**清空 / 删除**类操作必须用 `confirmDanger()` 打字校验对话框，不能用 `window.confirm`。

## 理由

**用户视角**：

- 家长就算手抖、孩子乱点都不会丢数据
- 输入"清空" / "删除"两个字的过程让用户**有时间反悔**

**工程视角**：

- 一次性写好，所有危险操作复用
- 与现有 toast / loading overlay 设计语言统一
- 比"二次确认 + 倒计时"之类花哨方案简单可靠

## 与 AI 讨论摘要

- Codex finding 里同时点出 P1 删除单事务原子性问题（`deleteWeek` 多 store 事务被拆开），同步合并为单事务（多 store readwrite，详见 `vocab-review/js/db.js`）
- AI 顺手把 `withTransaction` 的"事务回调内只能 await IDB 请求"约束写进 `shared/db.js` JSDoc
