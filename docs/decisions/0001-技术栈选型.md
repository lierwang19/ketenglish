# 0001 — 技术栈选型：原生 JS + IndexedDB + PWA

- **日期**：2026-04-15
- **状态**：✅ 生效中
- **影响范围**：全局

## 背景

要做两个轻量学习工具（单词复习 + 听力精听），用户是家庭场景，主力设备是手机和平板。需要离线可用、数据持久化、零部署门槛、能 PWA 安装到主屏。

不需要服务端、不需要协作、不需要账号体系。

## 选项

| 方案 | 优 | 劣 |
|---|---|---|
| A. 原生 HTML/CSS/JS + IndexedDB + Service Worker | 零依赖、零构建、随便一台静态服务器就能跑、加载快 | 没有组件复用机制，UI 多了得自己抽 |
| B. React + Vite + IndexedDB | 组件化、生态、TypeScript 加成 | 构建链、依赖更新负担、bundle 体积、对家庭场景过重 |
| C. Vue / Svelte | 同 B，差不多 | 同 B |
| D. Tauri / Electron 桌面 App | 体验最好 | 完全不符合"家长在手机上随手用"的场景 |

## 决定

选 **A：原生 HTML + CSS + JavaScript（ES Module）+ IndexedDB + Service Worker**。

- 不引入任何前端框架
- 不引入构建工具（无 Vite / webpack / babel）
- 第三方库走 vendor 目录手动引入（如 OCR 用 tesseract.js）
- 部署用 Docker + nginx 静态托管

## 理由

**用户视角**：

- 加载快、离线可用、可装到主屏。家长打开就用，不用等。
- 数据本地，家庭隐私安全，不依赖网络。

**工程视角**：

- 项目规模可控（每个子系统 ≤ 10 个 JS 文件），不需要框架的组件抽象成本
- 零构建意味着任何人 clone 下来用 `python3 -m http.server` 就能跑
- 没有依赖更新的长期维护负担
- 三年后回来改代码，原生 JS 不会"过时"

## 与 AI 讨论摘要

- AI 提醒过：没有 TypeScript 后期重构会麻烦。回应：项目规模小、模块边界清晰，靠 JSDoc 注释 + 严格命名约定足够
- AI 提醒过：没有组件化，UI 重复代码会膨胀。回应：先观察，到 V2.0 真撑不住再考虑引入轻量方案（如 lit-html）
