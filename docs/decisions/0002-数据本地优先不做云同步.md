# 0002 — V1.0 数据本地优先，不做云同步

- **日期**：2026-04-15
- **状态**：✅ 生效中
- **影响范围**：全局

## 背景

学习数据（单词进度、错词、统计）需要持久化。是否要做账号 + 云同步，是个早期就要定的方向题，影响 schema 设计和成本结构。

## 选项

| 方案 | 优 | 劣 |
|---|---|---|
| A. 纯本地 IndexedDB | 零成本、零运维、隐私无压力、离线可用 | 换设备要手动导出导入；浏览器清缓存数据丢失 |
| B. 本地 + 云同步（自建 Supabase / Firebase） | 多设备、防丢 | 需要账号体系、运维成本、隐私合规、对家庭项目过重 |
| C. 本地 + 文件级备份导出（手动 JSON / 网盘） | 零成本、能手动防丢 | 不自动 |

## 决定

**V1.0 选 A，预留 C 的导出导入入口**。

- 所有数据存 IndexedDB，schema 设计成 JSON 可序列化的扁平结构
- 设置页提供 **完整数据导出 / 导入**（JSON 文件）
- 不引入任何云服务

## 理由

**用户视角**：

- 家长不希望孩子的学习数据上云
- 单人单设备使用为主，多设备同步不是刚需
- 万一要换设备，导出 JSON → 新设备导入即可

**工程视角**：

- 不引入账号 = 不引入注册 / 登录 / 重置密码 / OAuth / 隐私协议这一整套
- 不引入云 = 不引入后端 / 数据库 / API / 部署 / 监控 / 计费
- 单人项目维护负担可控
- IndexedDB schema 改动通过 onupgradeneeded 平滑迁移即可

## 与 AI 讨论摘要

- AI 提醒过：浏览器隐私模式 / 清缓存会丢数据。回应：在设置页明确告知，并强制定期导出备份提醒（V1.1 加）
- AI 提醒过：iOS Safari 对 IndexedDB 配额有限制。回应：监控 `navigator.storage.estimate()`，超过 80% 提示用户

## 状态

V2.0 如果要做家庭多设备共享，可能演进到 B（基于 CRDT 或简单的 Supabase Row Level Security）。届时新开 ADR。
