# 0003 — Service Worker 改为自动激活更新模型

- **日期**：2026-04-26
- **状态**：✅ 生效中
- **影响范围**：vocab-review（listening-player 后续应同步）

## 背景

V1.0 早期实现了"经典 SW 更新模型"：新 SW 装好后停在 `waiting` 状态，页面顶部弹"发现新版本，刷新后生效"横幅，用户点了才 `skipWaiting`。

实测发现两个问题：

1. **横幅不出现** —— 旧 app.js 把 `SKIP_WAITING` 发给了 `controller`（旧 active SW）而不是 `reg.waiting`（新 SW），消息无效
2. **用户体验差** —— 即使横幅正常出现，强迫家长理解"立即刷新"按钮的含义、要点击才能拿到新版本，对家庭学习场景过度复杂

修复横幅 bug 后，用户的 push back 是："为何需要手动卸载？没有别的办法吗，这样用户会很麻烦。"

## 选项

| 方案 | 优 | 劣 |
|---|---|---|
| A. 修横幅 bug，保持"用户点确认"模型 | 安全：用户主动控制版本切换 | 家长根本不该理解"SW waiting"概念；横幅本身是噪音 |
| B. SW 自动激活：`self.skipWaiting()` + `clients.claim()` + 页面 `controllerchange` 自动 reload | 用户无感升级，零操作 | 万一新版本有 bug，用户被"强制升级"；需要保证 SW 更新不会破坏已存数据 |
| C. 经典 + 简化提示（仅小红点） | 折中 | 还是要用户操作 |

## 决定

选 **B：SW 自动激活模型**。

落地：

- `sw.js` install 阶段调 `self.skipWaiting()`，新版装好立即跳过 waiting
- `sw.js` activate 阶段调 `clients.claim()`，立即接管所有页面
- 页面 `app.js` 监听 `controllerchange`，触发后自动 `location.reload()`（一次性防抖）

## 理由

**用户视角**：

- 家长 / 孩子学习场景不是银行 App，不需要"等用户点确认"的版本控制
- 用户**只感受到**：刷新一下，新功能就有了。零认知负担

**工程视角**：

- 用户数据全部存页面侧 IndexedDB，SW 只缓存静态资源；SW 切版本不影响数据
- IndexedDB schema 升级走 `onupgradeneeded` 由页面侧执行，与 SW 无耦合
- 部署节奏可控（小作坊单人开发，不会一天多版本）

**风险对冲**：

- `CACHE_NAME` 必须每次 bump（`v10` → `v11`），保证旧缓存被清掉
- ADR / CONVENTIONS 强调"任何 cache 资源变化必须 bump 版本号"

## 与 AI 讨论摘要

- 最初实现的是 A 方案。Codex 复核 4 个 review finding 后修了一波，但用户横幅仍不弹（bug 是 SKIP_WAITING 发错对象）
- 修 bug 时 AI 提议用 DevTools 手动 unregister 旧 SW 来测试。**用户 push back**：这种调试手段不能扔给真实家长用户
- 切换到 B 方案，AI 担忧"强制升级"风险，用户决定接受（小项目、单人开发、数据隔离已保护）

## 状态

无后续修订。`listening-player/sw.js` 应在下次开发时同步该模型。
